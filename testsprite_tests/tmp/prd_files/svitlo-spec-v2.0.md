# SVITLO

**Power Generator Maintenance Application**

**Product Specification — MVP — Version 2.0**

## 1. Overview

Svitlo is a web application for tracking the usage and maintenance of power generators. It targets both individual users managing personal generators and businesses managing generators across multiple locations.

The core problem is the absence of a structured system for knowing when a generator needs maintenance, whether it has been run too long without rest, and who last started or stopped it. Svitlo solves this with a simple start/stop session log, maintenance templates that trigger by hours or by calendar time, and role-based access for administrators and employees.

Svitlo uses a local-first approach. All reads and writes happen against a local database on the device, so the essential features — starting and stopping generators, logging maintenance, viewing history — work without an internet connection. Data synchronizes to the server in the background when connectivity is available. Full offline operation has one inherent limitation: changes made by one user are not visible to another until both devices have synced.

## 2. Users, Organizations & Roles

### 2.1 Authentication

Two sign-in methods are supported:

- **Email and password.** Users register with an email address and a password. Standard email verification is required before the account is fully active.
- **Google OAuth (Sign in with Google).** Users may register and sign in using their Google account.

Both methods are handled by BetterAuth. A user who registers with one method and later signs in with the other will be linked to the same account if the email address matches.

### 2.2 Organizations

- Any authenticated user may create an organization. The creating user automatically becomes its sole administrator.
- An organization has exactly one administrator. This cannot be changed in the MVP.
- An organization may have any number of employees.
- A user may simultaneously be the administrator of multiple organizations and an employee in others.
- There is no location, store, or site entity. A generator's physical location is recorded as free text in the generator's own description field.

### 2.3 Roles & Permissions

A user's role is derived from the data, not stored as a column. If a user's ID matches `organizations.admin_user_id` they are the administrator of that organization; if they appear in `organization_members` they are an employee.

| Role | What they can do |
|------|-----------------|
| Administrator | Create and edit generators. Create and edit maintenance templates. Invite and remove employees. Assign employees to generators. Log maintenance records. View all history across the organization. Generate AI-suggested maintenance plans. |
| Employee | Start and stop sessions on generators they are assigned to. Log maintenance records on generators they are assigned to. View generator state and maintenance history for their assigned generators. |

### 2.4 Invitations

- The administrator invites a new employee by entering their email address. The invitation appears inside the app for any user whose account matches that email address.
- If no account with that email exists yet, the invitation will appear automatically once a user signs in with a matching email address.
- The same email address cannot be invited to the same organization more than once at a time.
- When the invitee accepts: a member record is created and the invitation row is deleted.
- When the invitee declines or the administrator cancels: the invitation row is deleted.
- Invitations have no expiry. They persist until acted upon.

## 3. Database Schema

The following tables make up the application's data model. Several values visible in the UI are not stored as columns but are computed at query time from the session log; these are noted explicitly and defined in section 4.

### 3.1 users

This table is created and managed entirely by the BetterAuth library. The application never writes to it directly. The columns below are the ones relevant to application logic; BetterAuth adds additional internal columns not listed here.

| Column | Type & Notes |
|--------|-------------|
| id | TEXT — primary key, generated by BetterAuth |
| name | TEXT — display name from the OAuth provider or entered at registration |
| email | TEXT — email address from the OAuth provider or entered at registration |
| emailVerified | BOOLEAN — whether the email has been verified |
| image | TEXT — profile photo URL from the OAuth provider, nullable |
| createdAt | TIMESTAMPTZ |
| updatedAt | TIMESTAMPTZ |

> ℹ️ Column names follow BetterAuth's camelCase convention in the underlying table. The application queries this table read-only and references `users.id` as a foreign key throughout the schema.

### 3.2 organizations

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| name | TEXT |
| admin_user_id | TEXT — FK → users |
| created_at | TIMESTAMPTZ |

### 3.3 organization_members

Contains only employee rows. The administrator is identified by `organizations.admin_user_id` and does not have a row here.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| organization_id | UUID — FK → organizations |
| user_id | TEXT — FK → users |
| joined_at | TIMESTAMPTZ |

> ℹ️ (organization_id, user_id) is unique.

### 3.4 invitations

Contains only outstanding (unanswered) invitations. Rows are deleted on acceptance, decline, or cancellation. There is no status column and no history of past invitations.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| organization_id | UUID — FK → organizations |
| invitee_email | TEXT |
| created_at | TIMESTAMPTZ |

> ℹ️ (organization_id, invitee_email) is unique.

### 3.5 generators

Generators belong directly to an organization. Physical location, model number, serial number, and any other metadata are stored as free text in the description field.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| organization_id | UUID — FK → organizations |
| name | TEXT — e.g. "Back Yard Honda" or "Warehouse 3 — Diesel" |
| generator_type | TEXT — free text, e.g. diesel, gas, propane, petroleum |
| description | TEXT — nullable, free text for location, model, serial number, notes |
| max_consecutive_run_hours | REAL — maximum hours the generator may run in one continuous streak before rest is required |
| required_rest_hours | REAL — minimum hours the generator must rest once the run limit is reached |
| run_warning_threshold_pct | INTEGER — percentage of max_consecutive_run_hours at which a warning is issued; default 80 |
| created_at | TIMESTAMPTZ |

> ℹ️ There is no status column. Generator state (running / resting / available) is always derived at query time from the session log. See section 4.2.

### 3.6 generator_user_assignments

Many-to-many between generators and users. An employee may be assigned to multiple generators; multiple employees may be assigned to the same generator. The administrator does not require an assignment row to access generator data.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| generator_id | UUID — FK → generators |
| user_id | TEXT — FK → users |
| assigned_at | TIMESTAMPTZ |

> ℹ️ (generator_id, user_id) is unique.

### 3.7 generator_sessions

Append-only log of every start/stop cycle. A session is open while `stopped_at` is NULL, meaning the generator is currently running.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| generator_id | UUID — FK → generators |
| started_by_user_id | TEXT — FK → users |
| stopped_by_user_id | TEXT — FK → users; NULL while session is open |
| started_at | TIMESTAMPTZ |
| stopped_at | TIMESTAMPTZ — NULL until the session is closed |

> ℹ️ `duration_hours` is not stored. It is always computed as (stopped_at − started_at).

> ℹ️ At most one session per generator may have `stopped_at IS NULL` at any given time. Enforced at the application layer and by a partial unique index on (generator_id) WHERE stopped_at IS NULL.

### 3.8 maintenance_templates

Defines a recurring maintenance task for a specific generator. No category types are enforced; the task name is free text. Only the administrator can create or edit templates.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| generator_id | UUID — FK → generators |
| task_name | TEXT — e.g. "Oil Change", "Air Filter", "Full Service" |
| description | TEXT — nullable, instructions or notes for the person doing the work |
| trigger_type | ENUM — hours \| calendar \| whichever_first |
| trigger_hours_interval | REAL — run-hours between recurrences; required when trigger_type is hours or whichever_first |
| trigger_calendar_days | INTEGER — days between recurrences; required when trigger_type is calendar or whichever_first |
| created_at | TIMESTAMPTZ |

> ℹ️ When trigger_type = whichever_first, both trigger_hours_interval and trigger_calendar_days must be set. The task is due when either condition is reached first.

### 3.9 maintenance_records

A log of maintenance work that has actually been performed. Records are created at the moment work is done, not pre-generated as future tasks. Any user assigned to the generator — administrator or employee — may create a record.

| Column | Type & Notes |
|--------|-------------|
| id | UUID — primary key |
| template_id | UUID — FK → maintenance_templates |
| generator_id | UUID — FK → generators |
| performed_by_user_id | TEXT — FK → users |
| performed_at | TIMESTAMPTZ — when the work was actually carried out; defaults to now but may be set to a past time |
| notes | TEXT — nullable |

> ℹ️ There is no status or due-date column on this table. Whether a task is due or overdue is a computed condition derived from this log. See section 4.3.

## 4. Business Logic

### 4.1 Generator Lifetime Hours

Total lifetime hours are never stored. They are always computed as the sum of (stopped_at − started_at) across all closed sessions for the generator. For a currently open session, elapsed time since started_at is included in the live total.

### 4.2 Generator State

The application derives one of three states for every generator at runtime. No state value is persisted.

| State | Derivation rule |
|-------|----------------|
| running | There exists a session for this generator where `stopped_at IS NULL`. |
| resting | The most recent closed session ended a consecutive run streak that met or exceeded `max_consecutive_run_hours`, AND (that session's `stopped_at` + `required_rest_hours`) > now. |
| available | Neither of the above conditions is true. |

**Consecutive Run Hours**

The current consecutive run hours are the sum of session durations since the last rest-triggering event. A rest-triggering event is the most recent closed session whose cumulative consecutive hours at close time met or exceeded `max_consecutive_run_hours`. All sessions after that session's `stopped_at` are part of the current streak.

If no rest has ever been triggered, all sessions since the generator was created form the streak.

**Rest End Time**

`rest_ends_at` = `stopped_at` of the rest-triggering session + `required_rest_hours`. This is a computed value; it is not stored.

**Warning Condition**

A warning is active when current consecutive run hours ≥ (`max_consecutive_run_hours` × `run_warning_threshold_pct` / 100). The default threshold is 80% and is configurable per generator by the administrator.

### 4.3 Session Rules

**Opening a Session**

- Precondition: generator state must be `available`. A session cannot be opened if the generator is `running` or `resting`.
- A new session row is inserted with `started_at` = now and `stopped_at` = NULL.

**Closing a Session**

- `stopped_at` is set to now on the open session row.
- Generator state is re-evaluated. If the new consecutive hours total meets or exceeds `max_consecutive_run_hours`, the generator enters the `resting` state.
- The session is never auto-closed by any server process. The decision to stop the generator belongs to the user.

### 4.4 Maintenance Due Date Computation

Because `maintenance_records` are a completed-work log rather than a pre-generated task list, the next due date for any template is always computed at query time from the most recent record for that template.

**Hours-based (trigger_type = hours)**

Next due when: lifetime hours at the time of the last record + `trigger_hours_interval` ≤ current lifetime hours. If no record exists yet, the task is due once the generator has accumulated `trigger_hours_interval` lifetime hours.

**Calendar-based (trigger_type = calendar)**

Next due on: `most_recent_record.performed_at` + `trigger_calendar_days`. If no record exists yet, the task is due `template.created_at` + `trigger_calendar_days`.

**Whichever-first (trigger_type = whichever_first)**

Both due dates are computed independently. The task is due when the earlier of the two conditions is met.

**Overdue**

A template is overdue when its computed next due date is in the past and no new `maintenance_record` has been logged since. This is a computed condition, not a stored status.

### 4.5 Employee Removal

When the administrator removes an employee from an organization:

- All `generator_user_assignments` rows for that employee within the organization are deleted.
- Each generator the employee was assigned to is automatically assigned to the administrator so no generator is left unmonitored.
- Any open session started by the removed employee remains open and must be closed manually by the administrator or another assigned user.

### 4.6 AI-Suggested Maintenance Plans

Svitlo provides an AI-powered feature that generates maintenance plan suggestions for a generator. This feature is available to administrators only.

**How It Works**

When the administrator creates or edits a generator, they may request AI-generated maintenance suggestions. The system uses an AI agent that:

1. Takes the generator's known attributes as input: `name`, `generator_type`, and `description` (which may contain the manufacturer, model number, and other identifying details).
2. Performs web research to find the manufacturer's recommended maintenance schedule, common maintenance intervals for that generator model, and general best practices for the generator's fuel type.
3. Returns a list of suggested maintenance templates, each including: a task name, a description of the work involved, a recommended trigger type (hours, calendar, or whichever_first), and the suggested interval values.

**User Interaction**

- The administrator triggers the suggestion by pressing a "Suggest Maintenance Plan" button on the generator detail or generator edit screen.
- The AI agent's results are presented as a list of draft templates that the administrator can review.
- For each suggestion, the administrator may accept it (which creates a `maintenance_template` row with the suggested values), modify it before accepting, or dismiss it.
- Suggestions are not persisted. They exist only in the UI during the review session. If the administrator navigates away without accepting, the suggestions are lost and must be regenerated.
- The feature requires an internet connection because the AI agent performs live web research. If the device is offline, the button is disabled with an explanatory message.

**Scope & Limitations**

- The AI suggestions are advisory. The administrator is always the final decision-maker and may adjust any values before saving.
- The quality of the suggestions depends on the specificity of the generator's attributes. A generator named "Honda EU2200i" with a description containing the model number will yield better results than one named "Small Generator" with no description.
- The AI agent does not have access to the generator's session history or existing maintenance records. It bases its suggestions solely on the generator's static attributes and publicly available maintenance information.

## 5. Resolved Design Decisions

| Decision | Resolution |
|----------|-----------|
| Default organization on sign-in | The last-used organization is automatically selected. The user may switch context at any time. |
| Behavior when run limit is exceeded mid-session | The session is never auto-closed. Stopping the generator is the user's decision. The UI displays a warning when the limit is reached. |
| Hours-based maintenance detection | Evaluated client-side from local data. No server-side evaluation of hours conditions. |
| Employee removal side-effects | Generator assignments transfer to the administrator. Open sessions remain open. |
| AI maintenance suggestions | Advisory only. Not persisted. Requires internet. Administrator reviews and accepts individually. |
